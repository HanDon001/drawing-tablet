# VoiceCanvas 纯语音绘图工具 - 系统设计文档 (基于通义千问)

## 1. 项目概述

| 字段 | 内容 |
|------|------|
| 项目名称 | VoiceCanvas (语音画布) |
| 核心定位 | 面向肢体不便及视力障碍用户的无物理交互绘图工具 |
| 技术范式 | ASR + LLM (通义千问) + Function Calling + Canvas 渲染引擎 |
| 设计理念 | 将大语言模型作为"认知大脑"处理模糊、复杂的自然语言，将前端引擎作为"运动器官"执行精准操作，通过完善的语音闭环反馈，填补视障用户视觉信息的缺失 |

## 2. 系统架构设计

系统采用"云端认知 + 本地执行"的混合架构，保障理解的准确性与渲染的实时性。

### 2.1 核心模块划分

#### 感知层

- **ASR (语音识别)**：Web Speech API 持续监听，将语音转为文本。
- **TTS (语音合成)**：Web Speech API 播报千问生成的自然语言回复，提供听觉闭环。

#### 认知层

- **上下文管理器**：维护对话历史与画布状态树，控制 Token 消耗。
- **通义千问引擎**：接收文本与上下文，通过 Function Calling 输出结构化绘图指令及自然语言回复。

#### 执行层

- **指令路由与校验器**：拦截大模型幻觉，校验参数合法性。
- **Canvas 对象引擎**：采用对象化绘图（保留每个图形的属性），而非像素化绘图，支持后续的修改与查询。
- **坐标转换器**：将语义方位（"左上角"）转化为画布绝对坐标。

### 2.2 数据流闭环图

```text
[用户语音] "在中间画个太阳，然后把它变大点"
   ↓
[ASR 模块] 文本："在中间画个太阳，然后把它变大点"
   ↓
[上下文管理器] 拼接 Prompt (系统指令 + 画布状态 + 对话历史 + 用户文本)
   ↓
[通义千问 API] (触发 Function Calling)
   ├─> 工具调用 1: draw_shape(shape="circle", color="红色", position="中间", tag="太阳")
   ├─> 工具调用 2: edit_shape(target_tag="太阳", scale="larger")
   └─> 文本回复: "好的，我在画布中间画了一个红色的太阳，并把它变大了。"
   ↓
[指令路由与校验] 依次执行绘图指令，更新画布状态树
   ↓
[Canvas 对象引擎] 渲染图形
   ↓
[TTS 模块] 播报："好的，我在画布中间画了一个红色的太阳，并把它变大了。"
```

## 3. 核心技术设计：基于通义千问的 Function Calling

传统语音绘图依赖正则提取，容错率极低。通义千问的 Function Calling 让系统具备了"理解意图+拆解步骤"的能力。

### 3.1 工具集定义

我们向千问注册以下核心工具（部分展示）：

```json
{

    {
      "type": "function",
      "function": {
        "name": "draw_shape",
        "description": "在画布上绘制一个形状",
        "parameters": {
          "type": "object",
          "properties": {
            "shape_type": {
              "type": "string",
              "enum": ["circle", "rectangle", "triangle", "line"]
            },
            "color": {
              "type": "string",
              "description": "颜色名称，如红色、蓝色"
            },
            "size": {
              "type": "string",
              "enum": ["small", "medium", "large"],
              "description": "大小"
            },
            "position": {
              "type": "string",
              "description": "位置，如：中心、左上角、右下角"
            },
            "tag": {
              "type": "string",
              "description": "给这个图形起个名字，方便后续指代，如：太阳、房子"
            }
          },
          "required": ["shape_type", "position"]
        }
      }
    },
    {
      "type": "function",

          "type": "object",
          "properties": {
            "target_tag": {
              "type": "string",
              "description": "要修改的图形的名字或指代词，如：太阳、刚才的圆"

              "type": "string"
            },
            "new_size": {
              "type": "string"
            },

              "type": "string"
            }
          },
          "required": ["target_tag"]
        }
      }
    },
    {
      "type": "function",
      "function": {
        "name": "describe_canvas",
        "description": "查询并描述当前画布上的内容，专为视障用户设计",
        "parameters": {
          "type": "object",
          "properties": {}
        }
      }
    }
```

### 3.2 Prompt 工程设计

为了确保千问完美适配残障场景，System Prompt 必须包含严格的角色设定和空间映射规则：

```text
你是一个名叫"小画"的语音绘图助手，专门帮助肢体不便和视力障碍的用户画画。

【核心规则】
1. 你的回复将被语音播报，请保持简短、口语化，充满温度。
2. 每次执行操作后，必须用自然语言确认你做了什么，让看不见的用户安心。
3. 当用户使用代词（如"它"、"刚才那个"）或模糊描述（如"再大点"、"挪边上"）时，你必须结合上下文推断其意图。
4. 模糊语义映射规则：
   - 颜色：太阳=红色，天空=蓝色，草地=绿色。
   - 位置：边上=随机边缘，中间=中心点，旁边=相对偏移。
5. 如果用户指令缺少参数，自行使用合理默认值，不要反问用户。例如"画个圆"默认在画布中心画一个中等大小的黑色圆。

【当前画布状态】
{canvas_state_json}
```

## 4. 针对残障人士的专项设计

### 4.1 视障用户：听觉闭环与空间描述

- **强制反馈**：大模型的每一次 Function Calling，都必须伴随文本回复。例如，调用 `edit_shape` 时，千问必须回复"我已经把红圆变大了一些"，绝不静默执行。
- **语义化查询**：视障用户询问"画布上有什么"，触发 `describe_canvas` 工具。前端将对象树转为文本传给千问，千问将其重组为空间叙事："画布中间有一个红色的大太阳，左下角有一棵绿色的三角形小树。"

### 4.2 肢残用户：消除精准度依赖

- **标签化系统**：系统在生成图形时，自动或根据指令赋予图形 tag（如"太阳"）。后续操作均通过 tag 索引（"把太阳移动一下"），彻底消除对鼠标点击选中图形的依赖。
- **模糊比例缩放**：不支持"放大50像素"这种精准指令，而是通过千问映射为 `scale: "larger"` 或 `scale: "much_larger"`，由前端引擎按比例（如 1.5x, 2x）放大，降低用户心智负担。

## 5. 性能与容错设计

### 5.1 双通道指令路由 (解决 LLM 延迟)

通义千问 API 存在网络延迟（1-3秒），对于高频、简单的控制指令，体验极差。

- **快通道（本地正则）**：针对"撤销"、"重做"、"清空"、"停止"等无语义歧义的指令，前端直接拦截执行，延迟 < 100ms。
- **慢通道（千问 API）**：针对"绘图"、"修改"、"查询"等需要理解的指令，走大模型。

### 5.2 Token 控制与上下文管理

- **画布状态压缩**：不把图形的坐标像素值发给千问，只发 `[tag: 太阳, shape: 圆, color: 红, position: 中间]`，大幅节省 Token。
- **滑动窗口**：仅保留最近 5 轮对话历史和当前画布上最近 10 个对象的状态，防止上下文爆炸导致 API 超时或报错。

### 5.3 大模型幻觉拦截

- **Schema 校验**：前端接收到千问的 Function Call 后，必须校验参数。若模型幻觉输出了 `draw_shape(shape="apple")`，前端拦截并降级，通过 TTS 播报："抱歉，我目前还不会画苹果，我可以画圆形来代替吗？"

## 6. 指令集实现状态与差异说明

### 6.1 完美实现 (得益于 LLM)

| 指令类型 | 用户说法示例 | 千问处理逻辑 | 状态 |
|----------|--------------|--------------|------|
| 复合指令 | "在左上角画个蓝方块，右下角画个红三角" | 拆解为两次 draw_shape 顺序调用 | ✅ 100% |
| 指代消解 | "把刚才的圆改大点" | 结合上下文提取 target_tag，调用 edit_shape | ✅ 100% |
| 模糊语义 | "画一个像天空一样的圆" | 推断 color: 天蓝色，调用 draw_shape | ✅ 100% |
| 视障查询 | "现在画布上有什么？" | 调用 describe_canvas，生成空间描述文本 | ✅ 100% |

### 6.2 未完成及原因说明

| 计划能力 | 状态 | 未完成原因说明 |
|----------|------|----------------|
| 自由画笔 (海龟绘图) | ❌ 舍弃 | 技术限制：自由画笔需要"向前走10步，左转，再走20步"的高频连续交互。大模型的流式响应延迟（1-3秒）导致画笔动作严重卡顿，体验极差。替代方案：引导用户使用几何图形组合拼接。 |
| 复杂相对距离感知 | ⚠️ 降级 | 空间映射难点：用户说"把圆往右上角挪一点"，千问能理解方向，但"一点"很难量化。当前采用固定步长偏移（如移动画布宽度的10%），无法做到精准的相对距离感知。 |
| 多对象层级关系 | ❌ 未实现 | Prompt 复杂度：涉及"把红圆放到蓝方块上面（遮挡关系）"，前端虽支持 zIndex，但在千问的 System Prompt 中描述层级逻辑极易导致模型混乱，V1.0 暂不开放此能力，按绘制先后顺序叠加。 |

## 7. 交互流程示例：视障用户画风景

```text
User: "小画小画，帮我画一幅夕阳风景。"
System (Thinking): 触发千问，拆解为三个动作。
System (TTS): "好的，我来为您画一幅夕阳风景。"
System (Action 1): draw_shape(shape="rectangle", color="绿色", position="底部", tag="草地")
System (Action 2): draw_shape(shape="circle", color="橙色", position="中间偏上", tag="夕阳")
System (TTS): "我已经在画布底部画了绿色的草地，在上方画了一个橙色的夕阳。"

User: "太阳再大一点，颜色再红一点。"
System (Thinking): 指代消解（太阳=夕阳），双重属性修改。
System (Action): edit_shape(target_tag="夕阳", new_size="large", new_color="红色")
System (TTS): "已将夕阳变大，并改成了红色。"

User: "现在画面上是什么样子？"
System (Thinking): 触发查询。
System (TTS): "目前画布底部有一大片绿色的草地，上方偏中间的位置有一个红色的大夕阳。"
```
